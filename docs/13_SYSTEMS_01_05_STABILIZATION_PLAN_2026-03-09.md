# خطة تثبيت الأنظمة 01-05 (بعد مراجعة التقارير)

تاريخ: 2026-03-09
النطاق المعتمد: الأنظمة 01 إلى 05 فقط.

## 1) نطاق العمل
- داخل النطاق:
- نظام 01: البنية المشتركة (Users, Roles, Permissions, Audit, Lookups).
- نظام 02: النواة الأكاديمية.
- نظام 03: الموارد البشرية.
- نظام 04: الطلاب.
- نظام 05: التعليم والدرجات.
- خارج النطاق حاليًا (يتم تجاهله في هذه الخطة):
- النظام المالي (07) وبقية الأنظمة 06-19 غير المنفذة فعليًا.

## 2) ما تم اعتماده من تقارير التحليل
- نقاط صحيحة ومعتمدة للتنفيذ:
- تحسين وضوح نموذج الهوية والصلاحيات والنطاق.
- تقليل اللبس بين Role-based access وData scope.
- سد فجوات العرض في الواجهة التي تخفي علاقة الموظف بحساب المستخدم.
- نقاط مؤجلة عمدًا:
- أي ملاحظة تخص الأنظمة غير المنفذة (خصوصًا المالي).
- أي تغيير معماري كبير خارج 01-05 في هذه المرحلة.

## 3) سبب اللخبطة الحالية (Root Cause)
- اللخبطة ليست فقط في قاعدة البيانات، بل في طبقة العرض والتشغيل:
- `Employee` يمثل الملف الإداري/الوظيفي في HR.
- `User` يمثل حساب الدخول للنظام.
- `Role` و`Permission` يمثلان صلاحيات الدخول والإجراءات (RBAC).
- `EmployeeTeachingAssignment` و`EmployeeSectionSupervision` يمثلان نطاق البيانات داخل السنة/الشعبة/المادة.
- النتيجة: الربط موجود بالكود، لكن ظهوره في الواجهات كان غير كافٍ، فظهر للمستخدم كأنه نظامان منفصلان.

## 4) نموذج التشغيل الرسمي (Source of Truth)
1. إنشاء الموظف في HR.
2. إنشاء مستخدم وربطه بالموظف.
3. إسناد دور أو أكثر للمستخدم.
4. إسناد تدريس (section/subject/year) للمعلم.
5. إضافة نطاق إشراف عند الحاجة (view/manage flags).
6. التطبيق يطبق الناتج النهائي: RBAC + Data Scope معًا.

## 5) التنفيذ المرحلي
### المرحلة A (تم البدء والتنفيذ)
- إظهار علاقة الموظف بحساب المستخدم داخل واجهة الموظفين.
- إظهار أدوار الحساب المرتبط لكل موظف.
- إضافة توضيح نصي داخل واجهات إسناد التدريس ونطاقات الإشراف بأنهما لا ينشئان حساب/دور تلقائيًا.

### المرحلة B (قيد التنفيذ)
- توحيد رحلة التشغيل في الواجهة:
- من بطاقة الموظف: الانتقال السريع إلى صفحة المستخدمين لربط الحساب.
- إضافة تحذير تشغيلي عندما يكون موظف لديه إسناد/نطاق بدون حساب مستخدم مرتبط.
- مراجعة ترجمة/وضوح المصطلحات في صفحات HR + التعليم والدرجات.

### المرحلة C (إغلاق الفجوات الرئيسية 01-05)
- مراجعة شاملة للـ lookup usage في HR/Students/Teaching.
- تقرير تطابق نهائي بين README + DDL + Schema + Seed.
- Smoke test تشغيلي للسيناريو الكامل (مدير -> مشرف -> معلم -> ولي أمر).

## 6) نتائج التنفيذ المنجز في هذه الدفعة
- Backend:
- توسيع استجابة `employees` لتشمل `userAccount` + الأدوار الفعالة للحساب المرتبط.
- Frontend:
- عرض حالة الحساب المرتبط لكل موظف (`لديه حساب / بدون حساب`).
- عرض البريد/اسم المستخدم للحساب المرتبط.
- عرض الأدوار المرتبطة بالحساب داخل بطاقة الموظف.
- إضافة توضيحات تشغيلية في شاشات:
- إسناد التدريس.
- نطاقات الإشراف.
- تمكين انتقال سريع من بطاقة الموظف إلى شاشة إدارة المستخدمين مع تمرير بحث مبدئي.
- إضافة تحذير مباشر في نماذج:
- إسناد التدريس.
- نطاقات الإشراف.
- عندما يكون الموظف المختار بدون حساب مستخدم مرتبط.

## 7) تعريفات مهمة للفريق
- الدور (Role): ماذا يستطيع المستخدم أن يفعل في النظام.
- النطاق (Scope): على أي بيانات يمكن تطبيق هذه الصلاحيات (أي شعبة/مادة/سنة).
- لا يكفي الدور بدون نطاق في أعمال التعليم اليومية.
- ولا يكفي النطاق بدون دور يسمح بالإجراء المطلوب.

## 8) مخرجات المتابعة القادمة
- إضافة مؤشر جودة بيانات تشغيلي للموظفين:
- موظف نشط + حساب مرتبط + دور مناسب + إسناد/نطاق مكتمل.
- ترتيب مراجعة صفحات System 05 المتبقية بنفس نمط التوحيد والترجمة.
