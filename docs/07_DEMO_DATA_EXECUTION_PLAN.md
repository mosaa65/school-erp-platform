# خطة تنفيذ بيانات Demo التشغيلية (System 03 + System 04)

## الهدف
تجهيز بيانات تشغيل واقعية للاختبار اليومي بدون تحميل النظام بوحدات غير مطلوبة الآن.

## ما سننفذه الآن
1. الموارد البشرية (الحد الأدنى التشغيلي):
- الموظفون.
- إسناد التدريس.
- نطاقات الإشراف.
- حسابات المستخدمين للموظفين مع الأدوار.
2. الطلاب (الحد الأدنى التشغيلي):
- الطلاب.
- أولياء الأمور.
- ربط الطالب وولي الأمر.
- قيود الطلاب.
- حضور الطلاب (عينات).
- كتب الطلاب (عينات منطقية مبنية على مواد الصف).

## ما لن ننفذه الآن
1. تقييمات الأداء.
2. المخالفات.
3. مواهب الموظفين.
4. تتبع دورات متقدم.
5. نظام 11 (محرك الجدول الذكي) كمولّد آلي.

## قرارات معمارية
1. `core seed` يبقى للبيانات الثابتة فقط.
2. `demo seed` يبقى للبيانات التشغيلية المتغيرة.
3. كل Seed يجب أن يكون idempotent:
- `upsert` حيث أمكن.
- أو `find + update/create` مع مفاتيح طبيعية ثابتة.
4. عدم إدخال بيانات تجريبية بصيغة `DEMO-*` في الكيانات الأكاديمية الأساسية.
5. اعتماد السنة الأساسية الحالية:
- `AY-2026-2027-1448H`.

## توزيع بيانات Demo (المقترح)
1. الموظفون:
- 1 مدير مدرسة.
- 2 مشرفين.
- 6 مربيين صفوف.
- 12 معلمًا موزعين على مواد الصفوف المستهدفة.
2. الشعب المستهدفة:
- 6 شعب (مثال: `grade-01(A/B)`, `grade-07(A/B)`, `grade-10(A/B)`).
3. الطلاب:
- 20 إلى 30 طالبًا لكل شعبة (الافتراضي 24).
4. أولياء الأمور:
- 1 ولي أمر لكل 1 إلى 2 طلاب (لدعم سيناريو الأشقاء).
5. قيود الطلاب:
- كل طالب له قيد نشط في السنة الأساسية الحالية.
6. كتب الطلاب:
- لكل قيد: كتب المواد الأساسية الفعلية للصف.
- `bookPart=MAIN` كافتراضي.
- `WORKBOOK` لبعض المواد (مثل العربي/الرياضيات) كعينة.
7. الحضور:
- 3 أيام عينة حديثة لكل قيد/مجموعة ممثلة.

## منطق الربط المهم
1. إسناد التدريس:
- يربط المعلم بـ `section + subject + academicYear`.
- يستخدم للـ data scope الموضوعي (المادة داخل الشعبة).
2. نطاقات الإشراف:
- ليس تكرارًا.
- يستخدم كصلاحية شعبة-مستوى (عرض الطلاب/إدارة الواجبات/إدارة الدرجات) حتى لو لم يكن هناك إسناد مادة مباشر.
3. قيود الطلاب:
- هي طبقة التسجيل السنوي/الشعبة.
- كل العمليات التشغيلية (حضور/كتب/درجات/واجبات) يجب أن تمر عليها.
4. كتب الطلاب:
- حاليًا `bookPart` نصي.
- سيتم ضبطه بقيم تشغيلية ثابتة في الـ seed (`MAIN`, `WORKBOOK`) لتقليل العشوائية.

## حسابات المستخدمين
1. حساب لكل موظف رئيسي:
- ربط `user.employeeId`.
- إسناد دور من الأدوار النظامية (`teacher`, `supervisor`, `class_supervisor`, `school_admin`, `employee`).
2. حسابات أولياء الأمور:
- المرحلة الحالية: إنشاء حسابات مبدئية فقط بدون صلاحيات تشغيلية حساسة.
- السبب: لا يوجد بعد ربط مباشر `user <-> guardian` في النموذج الحالي.
- الربط الصريح لأولياء الأمور ينجز لاحقًا مع نظام تطبيق أولياء الأمور.

## التحقق بعد التنفيذ
1. `npm run prisma:migrate:deploy`
2. `npm run prisma:seed:core`
3. `npm run prisma:seed:demo`
4. Smoke checks:
- تسجيل دخول مدير/معلم/مشرف.
- المعلم يرى شعبه ومواده فقط.
- المشرف يرى نطاق الشعب المربوطة به.
- الطالب لا يظهر بدون قيد.
- كتب الطلاب مرتبطة بمواد الصف المقيد عليه.

## مخرجات نهائية مطلوبة
1. ملف موجز داخل `docs` يحوي:
- عدد السجلات التي تم إنشاؤها لكل كيان.
- قائمة حسابات الاختبار (البريد/اسم المستخدم).
- ملاحظات القيود الحالية (خصوصًا حسابات أولياء الأمور).
