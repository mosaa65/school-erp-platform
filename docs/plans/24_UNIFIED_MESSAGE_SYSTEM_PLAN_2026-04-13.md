# خطة نظام الرسائل الموحد على مستوى النظام

تاريخ الخطة: 2026-04-13  
النطاق: `frontend` بالكامل  
الهدف: بناء مكوّن رسائل موحد يستخدمه النظام كله، ويُدار من مكان واحد، ويمكن تخصيص مظهره وحركته وتموضعه من إعدادات الملف الشخصي.

## الهدف التنفيذي

إنشاء نظام رسائل موحد Elegant Notification System يحقق الآتي:

1. كل رسائل النجاح والخطأ والتحذير والمعلومة تستخدم نفس البنية البصرية.
2. كل الرسائل تستمد اللون والحجم والانبعاث والحركة من طبقة مركزية واحدة.
3. يمكن تغيير الشكل على كامل النظام من مكوّن واحد فقط.
4. يمكن للمستخدم تخصيص:
   - مكان الرسائل في الشاشة
   - نمط الحركة
   - سرعة الانبعاث
   - كثافة التأثير البصري
   - درجة اعتماد الرسائل على لون الثيم الحالي
5. أبسط استخدام للمطور يكون عبر API قصيرة جدًا تستقبل النص فقط.

## النتيجة المستهدفة

الرسالة يجب أن تبدو:

1. أنيقة جدًا بصريًا.
2. خفيفة وسريعة.
3. واضحة ومقروءة على الجوال والحاسوب.
4. متوافقة مع RTL.
5. ذكية في التمركز وعدم تغطية عناصر مهمة.
6. موحدة في النظام كله.

## المشكلة الحالية

الوضع الحالي متفرق وغير موحد:

1. بعض الرسائل تظهر كبانر داخل الصفحة.
2. بعض الرسائل تستخدم `FinanceAlert`.
3. بعض الرسائل نجاح/خطأ تُكتب يدويًا داخل كل شاشة.
4. لا يوجد نظام موحد للـ toast أو الـ floating feedback على مستوى النظام.
5. لا توجد إعدادات موحدة للمكان أو الحركة أو مدة الظهور.
6. الألوان الحالية ليست جزءًا من نظام رسائل شامل، بل مجرد styles موضعية.

## التشخيص الجذري

المشكلة ليست في لون الرسالة فقط.  
المشكلة أن النظام لا يملك Notification Layer موحدة.

اليوم يوجد:

- `FinanceAlert`
- بانرات نجاح/خطأ داخل بعض الشاشات
- رسائل inline

لكن لا يوجد:

- Message Provider عام
- Portal موحد للرسائل
- Motion system موحد
- API واحدة لإطلاق الرسائل
- User preferences تتحكم في الرسائل

## أنواع الرسائل المطلوبة

### 1. Success

تستخدم عند:

- الإنشاء
- الحفظ
- التحديث
- التفعيل
- نجاح عملية تشغيلية

اللون الافتراضي:

- أخضر

### 2. Error

تستخدم عند:

- فشل الحفظ
- فشل الطلب
- خطأ API
- خطأ تحقق غير قابل للإكمال

اللون الافتراضي:

- أحمر

### 3. Warning

تستخدم عند:

- اقتراب انتهاء صلاحية
- إجراء يحتاج انتباه
- نتيجة قابلة للاستمرار لكن فيها مخاطرة

اللون الافتراضي:

- أصفر/كهرماني

### 4. Info

تستخدم عند:

- إشعار عام
- معلومة تشغيلية
- تلميح مهم
- نتيجة ليست نجاحًا ولا خطأ

اللون الافتراضي:

- أزرق أو لون الثيم الهادئ

### 5. Neutral

تستخدم عند:

- رسائل وصفية
- تحديث حالة هادئة
- “تم فتح الوضع الفلاني”

اللون الافتراضي:

- محايد يميل للثيم

### 6. Loading / Progress

ليست رسالة تقليدية، لكنها مهمة جدًا في النظام:

- جارٍ الرفع
- جارٍ الحفظ
- جارٍ التحقق
- جارٍ التوليد

هذه يجب أن تكون ضمن النظام، لأن تجربة الرسائل لن تكتمل بدونها.

### 7. Actionable Message

رسالة فيها زر أو إجراء سريع:

- تراجع
- فتح السجل
- عرض التفاصيل

هذه ليست مطلوبة في كل الحالات، لكنها يجب أن تكون مدعومة من النظام.

## القرار المعماري

النظام المقترح يتكون من 5 طبقات:

### الطبقة 1: Message Tokens

قيم التصميم الأساسية:

- ألوان الرسائل
- درجات الحدود
- الخلفيات
- الظلال
- نصف القطر
- ارتفاع الرسالة
- الحد الأقصى للعرض
- مسافات النصوص والأيقونات
- سرعات الحركة

### الطبقة 2: Message Card

المكوّن البصري نفسه:

- الأيقونة
- النص
- زر الإغلاق
- شريط accent
- glow خفيف
- animation state

### الطبقة 3: Message Stack / Viewport

المسؤول عن:

- مكان الرسائل
- ترتيبها
- تكديسها
- منع التداخل مع الهيدر/الفوتر/FAB
- responsive behavior

### الطبقة 4: Message Service / Hook

API استخدام قصيرة جدًا:

```ts
notify.success("تم الحفظ بنجاح");
notify.error("تعذر حفظ البيانات");
notify.warning("تنتهي صلاحية الوثيقة خلال 3 أيام");
notify.info("تم تحديث النتائج");
```

القاعدة الأساسية:

- النص فقط هو input الإجباري.
- كل شيء آخر اختياري وله defaults مركزية.

### الطبقة 5: User Preferences

إعدادات من الملف الشخصي للتحكم في:

- مكان الرسائل
- أسلوب الحركة
- مدة الظهور
- مستوى التأثير البصري
- دمج الرسائل مع لون الثيم

## القرار الخاص بشكل الرسالة

الرسالة الموحدة يجب أن تحتوي بصريًا على:

1. أيقونة دلالية واضحة.
2. عنوان tone غير صريح في معظم الحالات.
3. نص واحد واضح ومقروء.
4. شريط accent أو halo صغير يحدد نوع الرسالة.
5. زر إغلاق أنيق وغير مزعج.
6. motion entry وexit جميل لكن ليس مبالغًا.

### النمط البصري المقترح

الرسائل لا تكون:

- مسطحة جدًا
- أو صاخبة جدًا
- أو كبيرة بشكل مزعج

بل تكون:

- glass-soft
- bordered
- accent-lit
- shadowed lightly
- highly readable

### مواصفات أولية

#### النجاح

- خلفية خضراء ناعمة جدًا
- حد أخضر شفاف
- أيقونة نجاح ناعمة
- glow بسيط

#### الخطأ

- خلفية حمراء هادئة
- حد أحمر أوضح
- أيقونة تنبيه/فشل
- اهتزاز خفيف اختياري عند الظهور إذا كان motion profile يسمح

#### التحذير

- خلفية كهرمانية هادئة
- أيقونة warning واضحة
- accent warm

#### المعلومة

- خلفية ثيمية باردة
- accent من لون النظام أو أزرق دلالي

## علاقة الرسالة بلون الثيم

### القاعدة

لكل رسالة لون دلالي افتراضي:

- نجاح: أخضر
- خطأ: أحمر
- تحذير: أصفر
- معلومة: أزرق/ثيمي

### التخصيص

يوجد خياران:

1. `Semantic First`
   - يحتفظ بلون الرسالة الدلالي
   - يستخدم لون الثيم فقط في اللمسات الثانوية

2. `Theme Blended`
   - يمزج tone الرسالة مع accent النظام
   - مثال:
     - النجاح يظل أخضر لكن border/glow يتأثر بلون المستخدم
     - info يصبح قريبًا من لون الثيم مباشرة

هذا أفضل من ربط كل الرسائل بلون المستخدم بالكامل، لأن الدلالة البصرية يجب ألا تضيع.

## الرسائل الثابتة مقابل العائمة

### 1. Floating Toast

لرسائل:

- الحفظ
- الإضافة
- النجاح السريع
- الخطأ القصير
- التحذير اللحظي

### 2. Inline Banner

لرسائل:

- أخطاء النموذج
- أخطاء الصفحة
- حالة فارغة
- تحذيرات تحتاج بقاء أطول

### القرار

الخطة لا تلغي الرسائل inline.  
بل تنشئ نظامًا بصريًا موحدًا يخدم:

- `Toast`
- `Inline Banner`
- `Section Notice`
- `Form Error Banner`

## نماذج الاستخدام المطلوبة

### أبسط استخدام

```ts
notify.success("تم حفظ السجل");
notify.error("تعذر حفظ السجل");
notify.warning("الحقل يحتاج مراجعة");
notify.info("تم تحديث العرض");
```

### استخدام متقدم اختياري

```ts
notify.success("تم حفظ السجل", {
  duration: 5000,
  actionLabel: "عرض",
  onAction: () => openDetails(),

```

لكن القاعدة المهمة:

- النص فقط يكفي.
- النظام يملأ الباقي من default config.

## الحركة والأنيميشن

الهدف ليس كثرة الحركة، بل جودة الحركة.

### Motion Entry

خيارات تصميمية:

1. Fade + Lift
2. Scale + Fade
3. Slide from edge
4. Soft spring

### Motion Exit

1. Fade out
2. Reverse slide
3. Shrink + fade

### توصية

الافتراضي:

- دخول: `fade + slight translate`
- خروج: `fade + collapse`

للتحذير والخطأ يمكن إضافة micro-emphasis:

- خطأ: نبضة قصيرة جدًا أو shake خفيف جدًا
- تحذير: pulse soft

لكن هذا يكون مرتبطًا بـ motion profile، وليس مفروضًا دائمًا.

## Responsive Behavior

### على الجوال

1. الرسائل تكون أعرض لكن لا تغطي عناصر التنقل المهمة.
2. `top-center` أو `bottom-center` غالبًا أفضل.
3. يجب احترام:
   - safe area
   - footer dock
   - FAB

### على الحاسوب

1. يسمح بتكديس 3 إلى 5 رسائل.
2. `top-right` أو `top-left` حسب RTL مناسب.
3. لا تتعدى عرضًا بصريًا مريحًا.

### القاعدة

العرض يجب ألا يكون ثابتًا بالكامل، بل responsive:

- `max-width`
- `min-height`
- `stack gap`
- `breakpoint-specific position`

## الرسائل الأخرى غير النجاح والخطأ والتحذير

نعم، يوجد أنواع مهمة أخرى:

1. `info`
2. `neutral`
3. `loading/progress`
4. `actionable`
5. `validation summary`
6. `empty-state notices`
7. `permission-denied notices`

لكن من ناحية tone visual العامة، يمكن ضبط النواة الأساسية على:

- success
- error
- warning
- info
- neutral

ثم يشتق منها الباقي.

## العلاقة مع ما هو موجود الآن

يوجد حاليًا نقطة انطلاق جيدة لكنها غير كافية:

- `FinanceAlert`

هذه يجب ألا تُلغى مباشرة، بل تتحول إلى جزء من النظام الجديد:

### المسار المقترح

1. نبقي `FinanceAlert` مؤقتًا.
2. نجعلها تعتمد لاحقًا على نفس tokens الخاصة بالنظام الجديد.
3. ننشئ بديلًا عامًا:
   - `InlineMessage`
4. ثم نرحل الشاشات إليه تدريجيًا.

## القرار الخاص بالتأكيدات

رسائل التأكيد مثل:

- حذف
- اعتماد
- تفعيل

ليست Toast.

هذه تبقى:

- dialog
- confirm sheet

لكن نتيجة التأكيد بعدها يجب أن تمر عبر نظام الرسائل الجديد.

## قواعد المحتوى

### النصوص

1. النص يكون مباشرًا وواضحًا.
2. لا تستخدم رسائل عامة جدًا مثل:
   - تم
   - نجحت العملية
3. الأفضل:
   - تم إنشاء السنة الأكاديمية
   - تعذر حذف السجل
   - تنتهي صلاحية الوثيقة خلال 3 أيام

### الطول

القالب يجب أن يدعم:

- رسالة قصيرة جدًا
- رسالة متوسطة
- رسالة طويلة نسبيًا مع سطرين أو ثلاثة

لكن يجب وضع قواعد:

1. لا يزيد الارتفاع تلقائيًا بشكل فوضوي.
2. بعد عدد أسطر محدد، يتحول إلى compact multi-line layout.

## قواعد التكديس

1. لا تظهر أكثر من 4 رسائل عائمة افتراضيًا.
2. الرسائل الجديدة تصعد فوق القديمة أو بالعكس حسب مكانها.
3. الرسائل المتشابهة يمكن دمجها اختياريًا.
4. الخطأ المتكرر لا يجب أن يملأ الشاشة.

### تحسين ذكي

إضافة deduplication:

- إذا جاءت نفس الرسالة 3 مرات خلال ثوانٍ قليلة، تعرض مرة واحدة مع عدّاد.

## مراحل التنفيذ

## المرحلة 1: تأسيس التصميم

المطلوب:

1. تعريف الأنواع `tone`, `variant`, `motionPreset`, `positionPreset`
2. تعريف message tokens
3. اعتماد palette افتراضية

### ناتج المرحلة

- ملف tokens
- ملف types
- مراجع التصميم البصري

## المرحلة 2: بناء المكونات

المطلوب:

1. `SystemMessage`
2. `SystemMessageStack`
3. `SystemMessageProvider`
4. `useSystemMessage`
5. `InlineMessage`

### ناتج المرحلة

نواة النظام الموحد للرسائل.

## المرحلة 3: ربطه بالملف الشخصي

المطلوب:

1. إنشاء preferences schema
2. حفظ الإعدادات
3. شاشة تحكم داخل profile
4. live preview

### ناتج المرحلة

المستخدم يستطيع تخصيص:

- اللون
- الحركة
- المكان
- المدة

## المرحلة 4: اعتماد النظام في الشاشات الجديدة

المطلوب:

1. كل نجاح/فشل جديد يمر عبر `notify`
2. منع الرسائل اليدوية الجديدة

## المرحلة 5: ترحيل النظام القائم

ترحيل تدريجي للمصادر التالية:

1. البانرات اليدوية
2. رسائل success/error داخل workspaces
3. `FinanceAlert` الموضعية عندما تكون floating scenario
4. بعض رسائل form success/error

## المرحلة 6: التوحيد النهائي

1. ربط inline banners بنفس tokens
2. تقليل الرسائل اليدوية
3. اعتماد checklist مراجعة

## معايير القبول

يعتبر النظام ناجحًا إذا تحقق الآتي:

1. الرسالة نفسها تتغير على كل النظام من مكان واحد.
2. success/error/warning/info لها ألوان افتراضية واضحة.
3. الرسائل متجاوبة مع أحجام الشاشات.
4. الرسائل لا تغطي الفوتر أو FAB أو عناصر أساسية.
5. الـ motion جميل وغير مزعج.
6. يمكن تخصيص السلوك من الملف الشخصي.
7. استخدام المطور بسيط:
   - `notify.success("...")`
   - `notify.error("...")`
8. عند تعديل tokens أو component يتغير الشكل على مستوى النظام كله.

## Checklist للمراجعة

قبل اعتماد النظام:

1. هل tone الرسالة واضح؟
2. هل الرسالة مقروءة على الجوال؟
3. هل المسافات متوازنة؟
4. هل اللون الدلالي واضح لكن غير صارخ؟
5. هل الحركة أنيقة وغير مرهقة؟
6. هل الإغلاق اليدوي واضح؟
7. هل تم احترام RTL؟
8. هل تم احترام safe areas؟
9. هل يوجد fallback بدون حركة؟
10. هل النص الطويل لا يكسر التصميم؟
11. هل تخصيصات الملف الشخصي تنعكس فعليًا؟

## قرار مهم

هذه الخطة توصي بإنشاء **Notification Design System** وليس مجرد Toast Component.

الفرق كبير:

- الـ Toast فقط يحل “أظهر رسالة”
- أما النظام المقترح فيحل:
  - الرسالة
  - الـ tone
  - المظهر
  - الحركة
  - التجاوب
  - الإعدادات
  - التوحيد
  - سهولة الاستخدام

## الخلاصة

المطلوب الصحيح للنظام ليس:

- مكوّن رسالة جميل فقط

بل:

- طبقة رسائل كاملة
- قابلة للتخصيص
- موحدة بصريًا
- مركزية
- أنيقة جدًا
- وتُدار من إعدادات الملف الشخصي

والصيغة التنفيذية الأنسب هي:

1. Message Tokens
2. Message Component
3. Message Provider
4. Simple Notify API
5. Profile Preferences
6. Gradual Migration

## اسم مقترح للنظام

يمكن اعتماد اسم داخلي واضح:

- `System Message`
- أو `Unified Feedback System`
- أو `Elegant Notification Layer`

الاسم البرمجي المقترح:

- `system-message`

لأنه واضح، عام، ومناسب للتوسع لاحقًا.

## الخطة التنفيذية التفصيلية

### المرحلة 1: تأسيس النواة المشتركة

يتم إنشاء طبقة الرسائل المركزية أولًا قبل ترحيل أي شاشة:

1. تعريف `types` الخاصة بالرسائل:
   - `tone`
   - `variant`
   - `position`
   - `motionPreset`
   - `durationPreset`
   - `densityPreset`
2. تعريف `tokens` الموحدة:
   - الألوان
   - الحواف
   - الظلال
   - الـ blur
   - الـ spacing
   - أحجام الأيقونات
   - سرعات الحركة
3. تعريف config افتراضي عام للنظام.

### المرحلة 2: بناء طبقة العرض

يتم بناء المكوّنات المشتركة التالية:

1. `SystemMessageCard`
2. `SystemMessageInline`
3. `SystemMessageStack`
4. `SystemMessageViewport`
5. `SystemMessageProvider`
6. `useSystemMessage`
7. `systemMessageService`

### المرحلة 3: ربط إعدادات الملف الشخصي

يتم إنشاء قسم جديد في الملف الشخصي خاص بالرسائل:

1. موضع الرسائل
2. الحركة
3. مدة الظهور
4. درجة الثراء البصري
5. طريقة مزج اللون مع الثيم
6. الإيماءات
7. وضع تقليل الحركة

### المرحلة 4: اعتماد النظام في العمليات الجديدة

من هذه المرحلة فصاعدًا:

1. أي شاشة جديدة تستخدم `notify`.
2. يمنع إنشاء banner أو toast يدوي جديد.
3. أي نجاح أو خطأ أو تحذير يمر عبر النواة المشتركة.

### المرحلة 5: ترحيل الشاشات الحالية

يتم الترحيل على دفعات:

1. رسائل الحفظ والإنشاء والتحديث والحذف.
2. رسائل أخطاء API العامة.
3. رسائل التحذير التشغيلية.
4. رسائل الحالة داخل الفورمات.
5. الرسائل الموضعية مثل `FinanceAlert` إن كانت من نفس الفئة البصرية.

### المرحلة 6: الإغلاق النهائي

بعد اكتمال الترحيل:

1. منع الأنماط القديمة.
2. إزالة الرسائل اليدوية المكررة.
3. تحويل أي wrapper قديم إلى adapter مؤقت أو حذفه.

## الإعدادات الافتراضية المقترحة

### الإعداد الافتراضي للنظام

1. الموضع:
   - الحاسوب: `top-right` مع مراعاة RTL
   - الجوال: `top-center`
2. النمط البصري:
   - `soft-glass`
3. الحركة:
   - `elegant`
4. مدة الظهور:
   - success: قصيرة
   - info: قصيرة إلى متوسطة
   - warning: متوسطة
   - error: طويلة نسبيًا
5. الإغلاق:
   - يدوي + إغلاق تلقائي
6. الحد الأقصى للرسائل الظاهرة:
   - الجوال: 2 إلى 3
   - الحاسوب: 4

### سلوك الرسائل الحرجة

بعض الرسائل لا يفضل أن تختفي بسرعة:

1. الخطأ المرتبط بفشل حفظ فعلي.
2. التحذير الذي يخص بيانات حساسة.
3. الرسائل التي تحتوي إجراءً سريعًا.

هذه يمكن أن تستخدم:

- مدة أطول
- أو تثبيت مؤقت حتى الإغلاق اليدوي

## سياسة الألوان

### القاعدة العامة

يحافظ النظام على دلالة اللون أولًا:

1. `success` أخضر
2. `error` أحمر
3. `warning` أصفر/كهرماني
4. `info` أزرق أو ثيمي
5. `neutral` رمادي/ثيمي هادئ

### الدمج مع لون النظام

لون المستخدم لا يجب أن يلغي دلالة الرسالة، لكن يمكن أن يؤثر في:

1. حدود الرسالة
2. الـ glow
3. الخلفية الثانوية
4. زر الإغلاق
5. شريط accent

### ما لا يجب فعله

لا يجوز أن تصبح:

1. رسالة الخطأ خضراء
2. رسالة النجاح حمراء
3. الرسائل كلها بنفس اللون بسبب الثيم

لأن ذلك يكسر المعنى الدلالي ويضعف الوضوح.

## سياسة الحركة والانبعاث

### المبادئ

الحركة يجب أن تكون:

1. قصيرة
2. ناعمة
3. مقصودة
4. غير مشتتة
5. قابلة للتقليل أو التعطيل

### Presets مقترحة

1. `calm`
   - دخول وخروج هادئ جدًا
2. `elegant`
   - الافتراضي
   - fade + lift
3. `lively`
   - أكثر حضورًا
   - spring خفيف
4. `minimal`
   - مناسب لتقليل الحركة

### الإيماءات

يدعم النظام:

1. السحب للإغلاق
2. النقر على زر الإغلاق
3. hover pause على الحاسوب
4. tap-safe behavior على الجوال

## سياسة المحتوى البرمجي

### API المطور

الاستخدام الأساسي يجب أن يكون بهذه البساطة:

```ts
notify.success("تم إنشاء السجل");
notify.error("تعذر حفظ البيانات");
notify.warning("البيانات تحتاج مراجعة");
notify.info("تم تحديث العرض");
```

### API النص فقط

لتحقيق مطلبك بأن المكوّن يستقبل نصًا فقط:

1. يتم توفير `notify.success(text)`
2. يتم توفير `notify.error(text)`
3. يتم توفير `notify.warning(text)`
4. يتم توفير `notify.info(text)`

وجميع:

- اللون
- الأيقونة
- المدة
- الحركة
- المكان
- الحجم

تأتي من config موحدة تلقائيًا.

### التخصيص المتقدم

يبقى موجودًا لكن ليس هو الاستخدام الأساسي:

```ts
notify.error("تعذر حذف السجل", {
  persistent: true,
  actionLabel: "إعادة المحاولة",
});
```

## قواعد الترحيل على مستوى النظام

### ما الذي يصبح موحدًا للجميع

إذا تم تغيير هذه العناصر، سيتغير النظام كله:

1. الشكل العام للرسالة
2. الألوان الدلالية
3. أحجام الرسائل
4. الحواف والظلال
5. الأيقونات الافتراضية
6. الحركة
7. أماكن العرض
8. مدة الظهور الافتراضية
9. سلوك التكديس

### ما الذي قد يبقى مختلفًا حسب الصفحة

1. نص الرسالة نفسه
2. نوع الرسالة
3. هل الرسالة inline أو floating
4. وجود action إضافي في الحالات الخاصة

### قواعد منع الأنماط القديمة

بعد اعتماد النواة الجديدة:

1. يمنع كتابة banner نجاح/خطأ يدوي داخل كل workspace بدون سبب خاص.
2. يمنع إنشاء toast محلي خاص بشاشة واحدة.
3. يمنع تكرار `FinanceAlert` كنمط جديد خارج سياقه.
4. أي رسالة جديدة يجب أن تستخدم `notify` أو `InlineMessage`.

## شاشات الترحيل ذات الأولوية

الأولوية الأولى:

1. الشاشات التي تنفذ `create/update/delete`
2. الشاشات التي تعرض أخطاء API متكررة
3. الشاشات التي تحتوي رسائل نجاح/فشل مكتوبة يدويًا

الأولوية الثانية:

1. رسائل الفورمات الداخلية
2. رسائل inline داخل الصفحات المالية
3. الرسائل المرتبطة بالبحث والفلترة والاستيراد

الأولوية الثالثة:

1. الرسائل الثانوية
2. الرسائل التوضيحية
3. wrappers القديمة

## معايير نجاح التصميم

النظام الجديد يجب أن يحقق الآتي بصريًا:

1. يبدو premium وليس generic.
2. لا يشبه toast افتراضي جاهز بدون هوية.
3. يحافظ على هوية الثيم الحالية للنظام.
4. واضح جدًا في معناه حتى دون قراءة طويلة.
5. جميل على الجوال والحاسوب معًا.
6. لا يشعر المستخدم بأنه مزعج أو صاخب.

## معايير نجاح الهندسة

النظام الجديد يجب أن يحقق الآتي تقنيًا:

1. نقطة تحكم واحدة.
2. Provider واحد على مستوى التطبيق.
3. Hook موحد للمطورين.
4. ملف tokens واحد أو طبقة tokens واحدة.
5. قابلية ترحيل تدريجي بدون كسر النظام.
6. سهولة إزالة الأنماط القديمة لاحقًا.

## الخلاصة التنفيذية النهائية

المطلوب ليس مجرد رسالة جميلة، بل **منظومة رسائل موحدة** تتحكم في:

1. اللون
2. الشكل
3. الحجم
4. الحركة
5. الانبعاث
6. المكان
7. الاستجابة للشاشات
8. الإعدادات الشخصية
9. طريقة الاستخدام البرمجية

والتنفيذ الصحيح هو:

1. بناء `System Message Layer`
2. ربطها بالثيم
3. ربطها بإعدادات الملف الشخصي
4. اعتماد `notify.*("text")` كواجهة الاستخدام الأساسية
5. ترحيل الرسائل الحالية تدريجيًا
6. منع الأنماط القديمة بعد اكتمال الترحيل
